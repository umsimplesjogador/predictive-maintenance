# Guia de Entrevista Técnica: Case Manutenção Preditiva (Senior Data Scientist)

Este documento foi forjado para ser sua base argumentativa técnica para defesa das metodologias aplicadas na entrega deste teste.

## 1. Por que o modelo treinou com Split Temporal (`sort_by('Cycle')`) e não `train_test_split` tradicional?
**Argumentativa:** "O comportamento de um maquinário operante degrada com o tempo. Prever o desgaste no passado usando inteligência treinada com dados do futuro é a definição exata de *Data Leakage* (Vazamento de Dados). O `train_test_split` tradicional da API scikit-learn embaralharia a linha do tempo, fazendo o modelo performar absurdamente bem num falso cenário mágico onde ele 'já viu' o maquinário quebrar no Clico 100 para tentar adivinhar se ele vai quebrar no Ciclo 50. Como profissional Senior, assumi o controle e cortei os 20% finais da linha do tempo rigidamente. Desse modo, o modelo apenas treina com falhas de '1999', para tentar inferir as mecânicas das falhas em '2000'."

## 2. Abordagem de Valores Missing (`ffill`)? Por que não usar Média ou Mediana?
**Argumentativa:** "Quando analisamos componentes sensoriais industriais (Vibração Y, Pressão), eles não oscilam linearmente num buraco de dados de 2 segundos. Se o sensor desliga, ele liga operando no exato mesmo momento de vibração ou temperatura imediatamente contíguo. Injetar a *Média* de um mês todo do sensor destruiria a onda temporal que eu criei para as _rolling mechanisms_. Fazer `Forward Fill` propaga o estado inercial real da máquina até o sensor retornar seu pulso."

## 3. Explicação do Balanceamento (Estratégia Anti-Imbalance)
**Argumentativa:** "A máquina é fisicamente robusta e a incidência real de anomalia pré-falha gira perto dos módicos 5%. Optar puramente pelo SMOTE criaria dados sintéticos fantasmas em torno destróicoes físicas complexas demais para o interpolador. Como Sênior, eu sei que algoritmos de árvore baseados em Gradiente (LightGBM, XGBoost) e Ensembles (Random Forest) conseguem resolver essa matemática assintótica nativamente usando o hiperparâmetro `scale_pos_weight` ou `balanced_weighting`. E foi dessa forma que eu amarrei os painéis do Optuna, forçando a matemática orgânica a penalizar erros contra a classe minoritária sem inventar clones artificiais do mundo real (SMOTE)."

## 4. Orquestração Multimodal com Optuna para Optar por LightGBM
**Argumentativa:** "O XGBoost costuma ser glorificado e a primeira opção instintiva de qualquer competidor no Kaggle. Mas para uma esteira em nuvem, ele pode ser muito denso ou demorado. Meu Optuna englobou as 3 famílias dominantes não por preciosismo: Random Forest, XGBoost e LightGBM. A avaliação foi baseada no *Precision-Recall AUC* (Área abaixo da Curva de Precisão-Revocação) e *F1-Score*, que são as métricas perfeitamente cirúrgicas para classes imensamente desbalanceadas (a métrica ROC-AUC clássica costuma ser um falso amigo e inflar otimismo nestes casos). O LightGBM sagrou-se vencedor porque sua topologia por 'Leaf-wise' conseguiu abraçar e convergir hiperparâmetros melhores para isolar os gatilhos no sensor VibrationY no limite dos 5% de falhas, com um custo computacional consideravelmente inferior de execução na etapa de modelagem."

## 5. Estratégia do MLflow
**Argumentativa:** "A escolha de configurar o `Tracking URI` do *MLflow* local via `sqlite` mas sem subi-lo para repositórios centrais foi feita pesando nas _Boas Práticas de Engenharia de MLOps_ modernas. Como se tratava de um artefato estrito via Webhook -> Container via API Rest, atrelar diretórios massivos `.db` para uma nuvem custa caro e é anti-padrão. Portanto, a tracking local se auto-deleta dos commits no Git via `.gitignore` - servindo unicamente ao Cientista Lab, que manda ao Pipeline CICD apenas os pacotes validados Sêniores do micro-projeto."
